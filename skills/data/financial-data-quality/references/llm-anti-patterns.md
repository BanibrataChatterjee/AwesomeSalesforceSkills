# LLM Anti-Patterns — Financial Data Quality

Common mistakes AI coding assistants make when generating or advising on Financial Data Quality in FSC.
These patterns help the consuming agent self-check its own output.

## Anti-Pattern 1: Advising Standard Duplicate Rules for FinancialAccount

**What the LLM generates:** Instructions to navigate to Setup > Duplicate Management > Duplicate Rules > New Rule and select `FinancialAccount` as the protected object, then create a Matching Rule targeting `FinancialAccount` fields such as `ExternalAccountNumber__c` or `AccountNumber`.

**Why it happens:** LLMs trained on generic Salesforce documentation know that Duplicate Rules and Matching Rules exist in Salesforce and that they can be configured for custom and standard objects. The FSC-specific limitation — that `FinancialAccount` is explicitly excluded from the standard Duplicate Management framework — is not prominently documented in high-volume training sources. The model generalizes from the pattern it knows (Duplicate Rules work on objects) without knowing the exception.

**Correct pattern:**

Standard Duplicate Rules cannot be configured for `FinancialAccount` (Core FSC) or `FinServ__FinancialAccount__c` (managed-package). Duplicate detection must be implemented as an Apex `before insert` trigger using a business key (external account number, owner + account type combination, or product-specific ID such as ISIN or policy number):

```apex
trigger FinancialAccountDuplicateCheck on FinancialAccount (before insert) {
    Set<String> keys = new Set<String>();
    for (FinancialAccount fa : Trigger.new) {
        if (fa.ExternalAccountNumber__c != null) keys.add(fa.ExternalAccountNumber__c);
    }
    if (keys.isEmpty()) return;
    Map<String, Id> existing = new Map<String, Id>();
    for (FinancialAccount ea : [SELECT Id, ExternalAccountNumber__c FROM FinancialAccount WHERE ExternalAccountNumber__c IN :keys]) {
        existing.put(ea.ExternalAccountNumber__c, ea.Id);
    }
    for (FinancialAccount fa : Trigger.new) {
        if (fa.ExternalAccountNumber__c != null && existing.containsKey(fa.ExternalAccountNumber__c)) {
            fa.addError('Duplicate FinancialAccount detected: ExternalAccountNumber ' + fa.ExternalAccountNumber__c + ' already exists.');
        }
    }
}
```

**Detection hint:** Look for instructions that reference `Setup > Duplicate Management` in the context of `FinancialAccount` or `FinServ__FinancialAccount__c`. Any such instruction is incorrect — standard Duplicate Rules do not cover these objects.

## Anti-Pattern 2: Writing a Non-Bulk-Safe Duplicate Trigger with SOQL Inside a For-Loop

**What the LLM generates:** An Apex trigger that places a SOQL query inside the iteration over `Trigger.new`, querying for existing records one at a time:

```apex
// WRONG — generated by LLM
trigger FinancialAccountDuplicateCheck on FinancialAccount (before insert) {
    for (FinancialAccount fa : Trigger.new) {
        List<FinancialAccount> existing = [
            SELECT Id FROM FinancialAccount
            WHERE ExternalAccountNumber__c = :fa.ExternalAccountNumber__c
            LIMIT 1
        ];
        if (!existing.isEmpty()) {
            fa.addError('Duplicate detected');
        }
    }
}
```

**Why it happens:** Many LLM training examples for Apex triggers demonstrate single-record logic that is easy to understand. The bulk-safe pattern (collect keys, one SOQL, Map lookup) is the correct Salesforce pattern but requires more code. Models often simplify to the readable one-record pattern without applying the governor-limit constraint.

**Correct pattern:**

```apex
// CORRECT — bulk-safe
trigger FinancialAccountDuplicateCheck on FinancialAccount (before insert) {
    Set<String> keys = new Set<String>();
    for (FinancialAccount fa : Trigger.new) {
        if (fa.ExternalAccountNumber__c != null) keys.add(fa.ExternalAccountNumber__c);
    }
    if (keys.isEmpty()) return;
    Map<String, Id> existingMap = new Map<String, Id>();
    for (FinancialAccount ea : [SELECT Id, ExternalAccountNumber__c FROM FinancialAccount WHERE ExternalAccountNumber__c IN :keys]) {
        existingMap.put(ea.ExternalAccountNumber__c, ea.Id);
    }
    for (FinancialAccount fa : Trigger.new) {
        if (fa.ExternalAccountNumber__c != null && existingMap.containsKey(fa.ExternalAccountNumber__c)) {
            fa.addError('Duplicate: ExternalAccountNumber ' + fa.ExternalAccountNumber__c + ' already exists.');
        }
    }
}
```

**Detection hint:** Look for `[SELECT ... FROM FinancialAccount WHERE ... = :fa.SomeField]` inside a `for (FinancialAccount fa : Trigger.new)` loop. Any SOQL query inside a for-loop over trigger records is the anti-pattern.

## Anti-Pattern 3: Assuming the FSC Rollup Batch Self-Heals or Has Native Alerting

**What the LLM generates:** Advice that the FSC rollup batch will automatically retry after failure, or that FSC provides a native alert mechanism for rollup batch failures that can be configured in Setup. Recommendations that a practitioner simply "check FSC Settings" to see if there are issues, without implementing external monitoring.

**Why it happens:** Salesforce platform batch jobs in Apex (AsyncApexJob) do support retry configuration. LLMs generalize this platform behavior to the FSC rollup batch, which is a separately managed FSC feature that does not share the same retry semantics as custom Apex batch classes. The absence of self-healing and native alerting for the FSC rollup batch is a specific FSC limitation not prominent in general Salesforce training content.

**Correct pattern:**

The FSC rollup batch does not retry automatically and does not produce a native alert on failure. Stale household KPI values persist indefinitely until a manual re-run is triggered. Monitor rollup health using a custom mechanism:

```
1. Create a Rollup_Audit__c record updated after each successful rollup batch completion
2. Deploy a scheduled Apex job that reads the Last_Successful_Run__c timestamp
3. Alert operations via email or Platform Event if the timestamp exceeds the expected batch interval
4. Document the manual re-run procedure: FSC Settings > Rollup Configuration > Run Rollups Now
```

**Detection hint:** Look for phrases like "the FSC rollup will retry automatically," "configure retry in FSC Settings," or "FSC will alert you if the rollup fails." These claims are incorrect.

## Anti-Pattern 4: Writing FinancialAccount Validation Rules Without a RecordType Guard

**What the LLM generates:** A validation rule formula for a field that is required only on a specific FinancialAccount type (e.g., insurance, investment) but without a RecordType condition — meaning the rule fires on all FinancialAccount records regardless of type:

```
// WRONG — no RecordType guard
AND(
    NOT($Permission.Bypass_FSC_Validation),
    ISBLANK(PolicyNumber__c)
)
```

**Why it happens:** When a practitioner describes a requirement like "insurance FinancialAccount records must have a PolicyNumber," LLMs sometimes generate a direct formula without the RecordType scoping because the requirement statement implies the context but does not explicitly state the formula structure needed. The model may also miss that FSC ships multiple FinancialAccount RecordTypes that would be incorrectly blocked by an unguarded rule.

**Correct pattern:**

```
AND(
    NOT($Permission.Bypass_FSC_Validation),
    ISPICKVAL(RecordType.DeveloperName, 'InsurancePolicy'),
    ISBLANK(PolicyNumber__c)
)
```

Always include `ISPICKVAL(RecordType.DeveloperName, 'TargetType')` as a guard for any FinancialAccount validation rule that applies only to a specific account type.

**Detection hint:** Look for `FinancialAccount` validation rule formulas that include `ISBLANK(SomeField)` without an `ISPICKVAL(RecordType.DeveloperName, ...)` guard when the requirement is type-specific. Also check: does the rule description mention "insurance" or a specific account type without the formula reflecting that scope?

## Anti-Pattern 5: Using Profile-Based Bypass Instead of Custom Permission for Integration Users

**What the LLM generates:** A validation rule formula that bypasses for system administrators or a specific profile:

```
// WRONG — profile-based bypass
AND(
    $Profile.Name != 'System Administrator',
    $Profile.Name != 'FSC Integration Profile',
    ISBLANK(ExternalAccountNumber__c)
)
```

Or hard-coding a user ID as the bypass condition.

**Why it happens:** Profile-based bypass is a common pattern in older Salesforce validation rule examples and tutorial content. LLMs that trained on pre-2020 Salesforce documentation frequently reproduce this pattern. The Custom Permission approach became the recommended pattern as integration architectures matured, but older patterns persist in training data.

**Correct pattern:**

```
AND(
    NOT($Permission.Bypass_FSC_Validation),
    ISBLANK(ExternalAccountNumber__c)
)
```

1. Create a Custom Permission named `Bypass_FSC_Validation`
2. Assign it via a Permission Set to integration service account users only
3. Use `NOT($Permission.Bypass_FSC_Validation)` in all FSC validation rules requiring bypass

**Why not profile-based:** Profile bypasses are too broad — a System Administrator bypass allows any admin to create invalid records, not just the integration service account. Profile names are fragile (renaming a profile breaks the bypass silently). Custom Permission bypass is auditable, least-privilege, and survives org restructuring without rule changes.

**Detection hint:** Look for `$Profile.Name` in any FSC validation rule formula, or for user ID comparisons like `$User.Id != '005...'`. Both are the anti-pattern. The correct pattern always uses `$Permission.SomeName`.

## Anti-Pattern 6: Assuming FinancialHolding Validation Rules Can Traverse Two Object Levels

**What the LLM generates:** A validation rule formula on `FinancialHolding` that traverses two relationship levels:

```
// WRONG — two-level traversal not supported in validation rule formulas
AND(
    ISPICKVAL(FinancialAccount.PrimaryOwner__r.ClientTier__c, 'Retail'),
    ISBLANK(RiskToleranceDocumented__c)
)
```

**Why it happens:** Salesforce formula fields and validation rules support cross-object references, and LLMs know this. The one-level limit for validation rule cross-object references is a specific constraint that is less visible in documentation than the general capability. Models generalize from "cross-object formulas work in validation rules" without applying the depth limit.

**Correct pattern:**

For checks requiring grandparent field access, use an Apex `before insert` / `before update` trigger on `FinancialHolding` with a SOQL join:

```apex
trigger FinancialHoldingValidation on FinancialHolding (before insert, before update) {
    Set<Id> accountIds = new Set<Id>();
    for (FinancialHolding fh : Trigger.new) {
        if (fh.FinancialAccountId != null) accountIds.add(fh.FinancialAccountId);
    }
    Map<Id, FinancialAccount> accounts = new Map<Id, FinancialAccount>([
        SELECT Id, PrimaryOwner__r.ClientTier__c
        FROM FinancialAccount
        WHERE Id IN :accountIds
    ]);
    for (FinancialHolding fh : Trigger.new) {
        if (fh.FinancialAccountId == null) continue;
        FinancialAccount fa = accounts.get(fh.FinancialAccountId);
        if (fa != null && fa.PrimaryOwner__r.ClientTier__c == 'Retail'
                && fh.RiskToleranceDocumented__c == null) {
            fh.addError('Risk tolerance documentation is required for retail client holdings.');
        }
    }
}
```

**Detection hint:** Look for validation rule formulas on `FinancialHolding` (or any child FSC object) that reference relationship traversal with two `__r` segments or two `.` levels. For example: `FinancialAccount.PrimaryOwner__r.SomeField` — the double traversal (`FinancialAccount` → `PrimaryOwner__r`) exceeds the one-level limit.
