# Gotchas — Data Cloud Vector Search Dev

Non-obvious Salesforce platform behaviors that cause real production problems in this domain.

## Gotcha 1: Chunk Size Is Not a Free Numeric Parameter in Easy Setup

**What happens:** A developer creates a Data Cloud vector search index using Easy Setup and later discovers they cannot change the chunk size, chunk overlap, or chunking strategy through the UI. The chunking parameters are automatically chosen by the platform and are locked at index creation time for Easy Setup indexes. There is no "Edit chunking" option after the fact.

**When it occurs:** Any time a developer uses Easy Setup expecting to tune parameters later, or any time the first index is created quickly for a proof-of-concept and then promoted to production without a rebuild.

**How to avoid:** Decide on chunking strategy before creating the index. If retrieval precision matters — or if the content type (FAQ, long-form documentation, technical procedures) suggests a specific chunk granularity — always use Advanced Setup from the start. If an Easy Setup index already exists and tuning is needed, delete the index and rebuild with Advanced Setup. The source DMO data is not affected by deleting the index.

## Gotcha 2: Query API Requires a Data Cloud Access Token, Not a Standard CRM Token

**What happens:** When a developer or external system calls the Data Cloud vector search Query API using an OAuth access token from a standard Salesforce CRM Connected App, the API returns `401 Unauthorized`. The two authentication systems are separate. The Data Cloud API endpoint uses a Data Cloud tenant token obtained from `/services/a360/token`, not the standard CRM `/services/oauth2/token` endpoint. Even a valid admin CRM session token is rejected.

**When it occurs:** Any time a developer reuses an existing CRM Connected App for Data Cloud API calls, assumes the standard token endpoint applies to all Salesforce APIs, or copies OAuth examples from CRM API documentation and applies them to Data Cloud endpoints.

**How to avoid:** Create a dedicated Data Cloud Connected App with the `cdpapi` OAuth scope. Obtain the Data Cloud token by posting to `<org-instance>/services/a360/token` with `grant_type=client_credentials` and the Data Cloud Connected App credentials. Note that the `instance_url` returned in the token response is the Data Cloud tenant URL (e.g., `https://<tenant>.c360a.salesforce.com`), which is different from the CRM instance URL and must be used as the base for all subsequent Data Cloud API calls.

## Gotcha 3: Einstein Trust Layer Masking Silently Drops Chunk Content Before the LLM

**What happens:** A retrieved chunk contains a DMO field that is classified as PII in the Data Cloud field taxonomy. The Einstein Trust Layer replaces the field value with a masking placeholder token before the chunk payload reaches the LLM. The chunk still occupies a top-K slot but delivers no useful content for the masked field. Developers see the agent appearing to ignore retrieved context without any explicit error — the Grounding tab in Agent Preview still shows the chunk as "retrieved," but the LLM-visible content is silently different from what the developer inspects in the raw DMO record.

**When it occurs:** When source DMOs contain fields that hold or reference customer PII (e.g., case descriptions referencing customer names, account numbers), and those fields were classified in the Data Cloud field taxonomy before indexing without the developer anticipating how masking would interact with retrieval.

**How to avoid:** Before building the vector search index, audit the source DMO fields against the Data Cloud field taxonomy. Identify any fields classified as Restricted or Personal. For fields that will be embedded, decide whether masking is acceptable at retrieval time or whether the field should be excluded from the indexed text. Review the Einstein Trust Layer audit log (`/services/data/v63.0/sobjects/AIGrndngEinsteinTrustLog`) for masking events during QA testing to confirm that masking behavior is deliberate rather than accidental.

## Gotcha 4: Changing the Embedding Model Requires Full Re-Indexing

**What happens:** A developer switches from the Salesforce-managed embedding model to a BYO model registered via Model Builder (or vice versa) after the vector index has been built. All existing vectors in the index were generated by the old model and are incomparable with vectors generated by the new model — vectors from different embedding models exist in different geometric spaces. Salesforce does not automatically re-embed existing chunks when the model is changed. The result is that search results degrade severely or become nonsensical because query embeddings from the new model are compared against index vectors from the old model.

**When it occurs:** When a team initially builds with the Salesforce-managed model for speed, then later decides to switch to a custom model for domain-specific accuracy, without planning for re-indexing downtime.

**How to avoid:** Treat embedding model selection as a deployment architecture decision, not a runtime setting. If a model change is necessary, plan for full index rebuild: deactivate the current index, update the embedding model setting, and trigger a full re-embed of all source chunks. For large corpora, this can take hours. Maintain the old index in parallel until the new index is fully built and validated.

## Gotcha 5: Near-Real-Time Refresh Requires Explicit Data Stream Configuration — Batch Is the Default

**What happens:** A developer configures a vector search index and expects newly published Knowledge articles (or updated DMO records) to appear in retrieval results within minutes. In practice, new records do not appear in search results until the next scheduled batch refresh window, which may be hours away. Data Streams default to scheduled batch mode; near-real-time continuous mode is not automatically enabled.

**When it occurs:** Any time a developer tests with freshly ingested source records and sees retrieval miss content they know is in the DMO. Also occurs in production when customer-facing agents fail to retrieve recently published content.

**How to avoid:** In the Data Stream configuration for the source object, explicitly set the refresh mode to continuous (near-real-time) if low-latency index currency is required. Verify the mode in Data Cloud Setup → Data Streams → the relevant stream → Refresh Settings. Document the chosen refresh cadence in the decision record, including the acceptable staleness window for the use case.
