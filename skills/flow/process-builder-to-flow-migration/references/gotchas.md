# Gotchas — Process Builder to Flow Migration

Non-obvious Salesforce platform behaviors that cause real production problems in this domain.

## Gotcha 1: Scheduled Actions Are Silently Cancelled on Process Builder Deactivation

**What happens:** When you deactivate a Process Builder process, all pending scheduled actions queued from that process are immediately cancelled. There is no warning dialog, no notification to users, and no automatic handoff to the replacement Flow.

**When it occurs:** Any time a Process Builder process with scheduled actions (e.g., "send email 5 days after Opportunity close") is deactivated — including during migration. If 200 Opportunities are in the "waiting" state expecting a scheduled follow-up email, those emails will never send.

**How to avoid:** Before deactivating, query `FlowInterview` records or `AsyncApexJob` to identify pending scheduled action work items. Plan cutover timing to minimize in-flight scheduled actions (e.g., migrate after a batch of closes completes). Communicate to stakeholders that pending scheduled actions will not fire after cutover.

---

## Gotcha 2: ISCHANGED() Criteria Silently Dropped or Incorrectly Mapped

**What happens:** The Migrate to Flow tool does not warn when it encounters `ISCHANGED()` or `ISNEW()` criteria in a Process Builder process. The generated flow may omit those conditions entirely (fire unconditionally) or generate a Decision condition that does not replicate the original behavior.

**When it occurs:** Any Process Builder process that uses `ISCHANGED(FieldName)` in entry criteria or action filter criteria.

**How to avoid:** Before running the tool, manually audit each process's criteria. Any process with ISCHANGED or ISNEW requires a manual rebuild, NOT the tool. After using the tool, always open the generated flow in Flow Builder and verify every Decision condition against the original process criteria.

---

## Gotcha 3: Active Process Builder + Active Flow on Same Object = Undefined Execution Order

**What happens:** Salesforce does not define or guarantee execution order when both an active Process Builder process and an active record-triggered flow target the same object. They execute in separate system sequences, and field values set by one may be overwritten or missed by the other.

**When it occurs:** During any migration window where the Flow is activated but the Process Builder process has not yet been deactivated — even for seconds.

**How to avoid:** Activate the Flow and deactivate the Process Builder process in the same change-set deployment or same manual change session. Never use a monitoring period where both are active simultaneously.

---

## Gotcha 4: Migrate to Flow Tool Cannot Be Scripted or Run via Metadata API

**What happens:** The Migrate to Flow tool is UI-only — there is no Metadata API equivalent, no Salesforce CLI command, and no way to run it programmatically in a CI/CD pipeline.

**When it occurs:** Teams try to automate mass migration of dozens of processes as part of a deployment script.

**How to avoid:** Accept that the tool requires manual UI interaction per process. For large-scale migrations, schedule dedicated admin time to run the tool per process. For processes requiring manual rebuild, those CAN be deployed via SFDX/source format as `.flow-meta.xml` files.

---

## Gotcha 5: Generated Flow Has No Fault Paths

**What happens:** Every flow generated by the Migrate to Flow tool omits fault paths on all DML elements (Update Records, Create Records). If a DML operation fails at runtime, the flow throws an unhandled fault error that emails the running user and rolls back the transaction — with no custom error message or logging.

**When it occurs:** Any time a migrated flow hits a DML error (e.g., validation rule block, record lock, field required error).

**How to avoid:** After running the tool, manually add Fault Path connectors to every DML element in the generated flow before activation. Route faults to a custom error message screen (for screen flows) or log to a platform event / custom object (for background flows). Reference `templates/flow/FaultPath_Template` for the canonical fault-handling pattern.
